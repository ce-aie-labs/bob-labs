## 문제

PR을 올리기 직전. 리뷰어가 지적할 것의 절반은 남의 시간을 쓰기 전에 스스로 찾을 수 있다. 새로운 눈으로 보기만 하면 되는데, 방금 그 코드를 쓴 사람에게 없는 게 정확히 그 새로운 눈이다.

## 프롬프트

**이건 챌린지 랩이다: 목표만 주고 프롬프트는 주지 않는다.** 직접 써서 돌려보고, 아래 기대 결과 체크리스트와 맞는지 확인할 것. 힌트는 막혔을 때만 열 것 - 핵심은 답이 아니라 직접 써보는 연습이다.

목표: 내 커밋 전 변경사항을 이 PR의 리뷰어라면 볼 방식으로 Bob에게 리뷰시키고, 그중 뭐가 실제로 머지를 막는 문제인지 알아내기.

<details markdown="1">
<summary>힌트 1 - 좋은 프롬프트의 뼈대</summary>

```
<정확히 무엇>을 <Bob이 어떤 역할이라고 가정할지>로서 리뷰해줘.

<신경 쓰는 관점을 우선순위 순으로> 봐줘.

각 이슈마다: <이슈 하나당 뭘 받고 싶은지>.

<듣고 싶지 않은 것>은 빼줘.
```

마지막 줄이 생각보다 중요하다. 없으면 포매팅 잔소리만 잔뜩 돌아온다.

</details>

<details markdown="1">
<summary>힌트 2 - 실제로 동작하는 프롬프트</summary>

```
커밋하지 않은 내 변경사항을, 네가 이 PR의 리뷰어라고 생각하고 리뷰해줘.

정확성과 엣지 케이스를 먼저 보고, 그다음 에러 처리, 그다음 이 레포의 다른
곳에서 이미 쓰고 있는 컨벤션을 어기는 부분을 봐줘.

각 이슈마다: 파일과 라인을 짚고, 실제로 뭐가 잘못되는지 설명하고,
머지 차단 / 고치는 게 좋음 / 선택 중에 등급을 매겨줘.

포매팅이나 스타일 잔소리는 빼줘. 그건 린터가 본다.
```

</details>

## 기대 결과

- [ ] 블로그에서 읽을 법한 일반론이 아니라, 특정 파일과 라인을 짚은 지적
- [ ] 각 항목의 등급 - 머지를 막는 것과 취향인 것을 구분할 수 있게
- [ ] 실제로 뭐가 깨지는지에 대한 설명 - "체크를 추가하는 걸 고려해보세요"가 아니라 "`items`가 비면 여기서 예외가 난다"
- [ ] 포매팅이나 import 순서 얘기는 없음 (프롬프트에서 제외했으니까)

## 팁

- 스타일 잔소리만 잔뜩 나오면 "포매팅은 빼줘" 지시가 없거나 약한 것. 빼고 싶은 걸 명시적으로 이름 붙여 말할 것.
- **커밋 전에** 돌릴 것. 변경사항이 아직 작업 트리에 있을 때는 고치는 비용이 싸지만, 리뷰가 시작된 뒤에는 비싸진다.
- 레포에 `AGENTS.md`가 있으면 Bob이 일반론이 아니라 우리 컨벤션 기준으로 리뷰한다. 없으면 `/init`부터 실행할 것.

## 응용

1. **브랜치 전체 리뷰**: "커밋 안 한 파일 말고, 이 브랜치가 `main` 대비 바꾼 것 전부를 리뷰해줘."
2. **관점 하나만**: "이 변경사항을 보안 관점으로만 리뷰해줘 - 인젝션, 인가, 그리고 시크릿을 로그에 남기는 부분."
3. **PR 설명으로 전환**: "...그리고 PR 설명을 써줘. 뭐가 바뀌었고 리뷰어가 어디를 유심히 봐야 하는지."
