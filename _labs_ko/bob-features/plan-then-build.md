## 문제

여러 계층(UI, API, 데이터베이스)에 걸친 기능을 추가해야 하는데, 에이전트에게 바로 코딩을 시키면 요청하지도 않은 파일들이 생기고 범위가 계속 불어난다. 코드 작성 전에 리뷰 가능한 명세를 한 단계 끼워넣는 멀티스텝 워크플로우.

## 프롬프트

이 랩은 프롬프트가 하나가 아니라 네 개다. 순서대로 실행하고, 각 단계 결과를 확인한 뒤 다음으로 넘어갈 것.

### 1단계 - Bob에게 프로젝트 컨텍스트 주기

**Agent 모드**에서 실행:

```
/init
```

**체크포인트:** Bob이 프로젝트 구조와 컨벤션을 담은 루트 `AGENTS.md`와 `.bob/` 아래 모드별 규칙 파일을 만든다. 루트 파일을 훑어보고, 프로젝트를 잘못 설명하고 있으면 지금 고칠 것. 이후 모든 단계가 이 파일 위에 쌓인다.

### 2단계 - 범위를 좁힌 계획 프롬프트 작성

**Plan 모드**로 전환(모드 선택기 또는 `/plan` 입력). 아래 템플릿을 채워서 실행. 네 개 블록이 계획을 조이는 역할을 하니, 빼면 두루뭉술한 계획이 나온다.

```
이 프로젝트에 <기능>을 추가하려고 해. <동작하면 사용자가 뭘 할 수 있는지 한 문장으로.>

범위:
- <계층 1 변경, 예: 예약 화면에 선택 UI 추가>
- <계층 2 변경, 예: API가 해당 값을 받아서 저장하도록 수정>
- <계층 3 변경, 예: DB 스키마에 컬럼 추가>
- <명시적으로 제외할 작업>은 하지 마. 범위 밖이야.

제약:
- 계획은 `plans` 폴더에 저장해줘
- 계획 파일은 3개를 넘기지 마
- <타협 불가 조건: 하위 호환성, 네이밍 컨벤션, 새 의존성 추가 금지 등>

완료 기준: <구체적이고 확인 가능한 조건>
```

**체크포인트:** Bob이 먼저 확인 질문을 할 수 있다. 답해주거나, 알아서 적당한 걸로 선택하라고 하면 된다. 그다음 `plans/` 폴더에 계획 파일을 쓴다. 아직 코드는 한 줄도 안 건드린 상태.

### 3단계 - 계획 리뷰하고 고치기

계획 파일을 열어서 다음 세 가지를 기준으로 읽을 것:

- **범위 일치** - 요청한 게 다 들어있나? 요청 안 한 게 끼어있진 않나?
- **모호한 표현** - "적절히 수정", "필요에 따라 추가" 같은 표현은 나중에 엉뚱한 코드가 되는 가정들이다.
- **파일 명시** - 각 단계가 건드릴 파일 이름을 짚고 있어야 한다. "예약 컴포넌트 수정"은 부족하다.

같은 Plan 모드 대화를 유지한 채로 발견한 걸 고친다:

```
계획에서 <구체적인 수정 사항>을 반영해서 업데이트해줘.
```

**체크포인트:** 수정된 계획이 실제 파일명을 짚고 있고 얼버무리는 표현이 없어야 한다. 그렇게 될 때까지 반복. 전체 워크플로우에서 실수를 잡는 비용이 가장 싼 지점이 여기다.

### 4단계 - 승인된 계획대로 구현

**새 대화**를 시작하고(플러스 아이콘), **Agent 모드**로 전환한 뒤 실행:

```
plans 폴더에 있는 계획대로 구현해줘
@plans/
```

**체크포인트:** Bob이 의도를 다시 추측하는 대신 계획을 따라 작업한다. 올라오는 도구 승인 요청을 승인하면 된다.

## 기대 결과

- [ ] 1단계 후, 일반적인 보일러플레이트가 아니라 *내 프로젝트*를 정확히 설명하는 루트 `AGENTS.md`
- [ ] 2단계 후, `plans/` 폴더의 계획 파일들 그리고 **아직 변경되지 않은 소스 코드** - 계획 단계는 되돌리기 쉽고 싸다는 게 핵심
- [ ] 각 단계가 내 레포의 실제 파일을 짚고 있고, 제외한 범위는 정말로 빠져있는 계획
- [ ] 4단계 후, 리뷰한 계획을 따라간 구현. "완료 기준"에 적은 조건이 실제로 충족됨

## 팁

- **4단계 전에 새 대화를 시작하는 건 그냥 정리 차원이 아니다.** 컨텍스트 윈도우를 비워서 구현이 계획 논의 전체와 토큰을 두고 경쟁하지 않게 하고, Bob이 "검토했던 것"과 "결정한 것"을 헷갈리지 않게 한다.
- 계획이 예상보다 훨씬 크게 나왔다면 범위 블록이 느슨했던 것. 구현 중에 통제하려 하지 말고 "~는 하지 마" 줄을 조일 것.
- 파일 개수 제약("계획 파일 3개 넘기지 마")은 실제로 효과가 있다. 상한이 없으면 계획이 아무도 안 읽는 문서로 불어난다.
- 계획만 필요하고 구현은 필요 없다면 3단계에서 멈춰도 된다. 리뷰된 계획은 그 자체로 쓸모있는 산출물이다. 팀원에게 넘기거나 티켓에 붙이면 된다.

## 응용

1. **기능 추가 대신 리팩토링**: 네 단계는 그대로, 2단계에서 목표 상태를 서술("800줄짜리 컨트롤러를 서비스 클래스로 분리, 동작 변경 없이")하고 완료 기준은 "기존 테스트 전부 통과"로.
2. **계획만 뽑아서 설계 문서로**: 3단계에서 멈추고 "승인된 계획을 이 코드베이스를 처음 보는 팀원용 설계 문서로 다시 써줘."
3. **계획을 바로 티켓으로**: 3단계 후 "계획의 각 단계를 완료 조건이 포함된 개별 이슈 설명으로 만들어줘."
