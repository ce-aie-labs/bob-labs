## 문제

새로 클론한 스프링 부트 레포지토리가 낯설어서, 첫 커밋 전에 기술 스택 · 패키지 구조 · 로컬 실행 방법 · 새 기능을 추가할 위치를 파악해야 하는 상황.

## 프롬프트

```
이 레포지토리를 설명해줘. 뭘 하는 프로젝트인지, 기술 스택은 뭔지, 코드가 어떻게
구성되어 있는지(주요 패키지/모듈과 각각의 역할), 로컬에서 어떻게 빌드하고
실행하는지, 그리고 새 기능(예: 새 REST 엔드포인트나 새 엔티티)을 추가하려면
어디를 봐야 하는지 알려줘.
```

## 기대 결과

- [ ] 앱이 뭘 하는지 한 문단 요약과 구체적인 기술 스택(프레임워크 버전, ORM, 템플릿 엔진, 데이터베이스) - 그냥 "스프링 부트 앱입니다" 수준이 아니라
- [ ] 실제 패키지 단위로 나눈 코드 구조 설명과 각각의 책임(엔티티 vs 리포지토리 vs 컨트롤러) - 일반적인 MVC 설명이 아니라
- [ ] 레포지토리에 있는 빌드 도구별 실제 빌드/실행 명령어, 그리고 실행 후 앱이 어디서 응답하는지
- [ ] 새 엔티티나 엔드포인트를 추가할 때 어디를 봐야 하는지에 대한 구체적인 안내, 참고할 기존 클래스를 콕 집어서

[spring-projects/spring-petclinic](https://github.com/spring-projects/spring-petclinic)으로 확인: 실제 기술 스택(Java 17, Spring Boot, Spring MVC, Spring Data JPA + Hibernate, Thymeleaf, H2/MySQL/Postgres)을 정확히 짚어냈고, 실제 패키지(`model`, `owner`, `vet`, `system`) 단위로 각 패키지의 클래스까지 짚어 코드 구조를 설명했으며, 실제 `./mvnw spring-boot:run` / `./gradlew bootRun` 명령어를 제시했고, 새 엔드포인트를 추가하는 패턴으로 `VetController`의 기존 REST 엔드포인트를 짚어줬다.

## 팁

- "컨트롤러, 서비스, 리포지토리 계층이 있다" 수준에서 멈추고 실제 클래스나 패키지 이름이 안 나오면, 코드를 제대로 읽지 않은 것 - "패턴 말고 실제 클래스와 파일 이름을 알려줘"라고 다시 요청.
- 빌드 도구가 두 개 이상인 레포(예: petclinic의 Maven + Gradle)에서는 엉뚱한 명령어를 제안받을 수 있음 - 실제로 존재하는 래퍼 스크립트(`mvnw`/`gradlew`)와 대조 확인.
- 큰 모노레포에서는 "이 레포지토리를 설명해줘"만으로는 최상위만 훑고 끝날 수 있음. "...`<패키지>` 패키지를 구체적으로 설명해줘"처럼 범위를 좁힐 것.

## 응용

1. **모듈 하나로 범위 좁히기**: "`<패키지>` 패키지만 설명해줘 - 뭘 담당하고 나머지 앱과 어떻게 연결되는지."
2. **온보딩 문서로 변환**: "...그리고 이 내용을 CONTRIBUTING.md에 넣을 짧은 온보딩 문서로 정리해줘."
3. **실제 작업과 엮기**: "...그리고 `<기능>` 엔드포인트를 추가하려면 구체적으로 어떤 파일을 건드려야 하는지 알려줘."
